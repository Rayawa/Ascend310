# X光YOLOv5s检测 - 昇腾310B 完整实现计划

## Goal
完成一个在 OpenHarmony 5.0.3 昇腾 310B 上运行的 X 光胸片检测应用：
- 用户在 UI 上传 X 光图片
- ArkTS 端将图片预处理为 640×640 的 RGBA 格式
- C++ 层通过 CANN ACL 加载 rawfile 中的 .om 模型并执行推理
- 对 YOLOv5s 输出 [1, 25200, 10] 做后处理（NMS），识别 5 类疾病
- 将检测框绘制在图片上返回给用户

## Scope / Non-goals

**包含：**
- 完整的 ArkTS 前端 UI（图片选择、预处理、推理状态、结果展示、检测框绘制）
- 完整的 C++ NAPI 桥接层（异步推理 + 结果转换）
- 完整的 C++ ACL 推理引擎（初始化、模型加载、推理执行、YOLOv5s 后处理 + NMS）
- .om 模型从 rawfile 拷贝到沙箱目录的自动处理
- 5 类疾病的标签与配色区分

**不包含：**
- .om 模型文件本身的训练/转换（假设已存在）
- DVPP 硬件加速预处理（采用 ArkTS 端软件预处理）
- 多图片批量推理
- 模型热更新/OTA

## Design

### 1. ArkTS 端预处理方案

**核心流程：**
1. 用户通过 PhotoViewPicker 选择图片
2. 使用 `image.createImageSource` → `createPixelMap` 解码
3. 使用 `PixelMap.scale()` 或重新创建 ImageSource + `DecodingOptions` 将图片缩放到 640×640
4. 使用 `PixelMap.readPixelsToBuffer()` 提取 RGBA ArrayBuffer
5. 将 640×640 RGBA ArrayBuffer 传给 C++ NAPI 接口

**关键决策：** 使用 `DecodingOptions` 中的 `desiredSize` 在解码时直接 resize，比 `PixelMap.scale()` 更高效（避免二次处理）。

### 2. C++ 层架构

**文件职责划分：**
- `hello.cpp`：纯 NAPI 桥接，负责参数提取、异步任务管理、C++ 结果 → JS 对象转换
- `yolo_infer.h/cpp`：纯 ACL 推理逻辑，不涉及 NAPI

**推理流程：**
1. `InitDevice()` — aclInit + aclrtSetDevice + aclrtCreateContext + aclrtCreateStream
2. `LoadModel()` — 从沙箱路径加载 .om，获取模型描述
3. `DoInference()` — 
   - 根据模型描述获取输入输出尺寸
   - RGBA→RGB 转换 + 归一化（/255.0）
   - Host→Device 内存拷贝
   - 创建 aclmdlDataset 并执行推理
   - Device→Host 取回输出
   - 调用 PostProcess()
4. `PostProcess()` — YOLOv5s 专属后处理
5. `Unload()` — 资源释放

### 3. YOLOv5s 后处理设计

**输出解析 [1, 25200, 10]：**
- 25200 = 3 个尺度 (80×80 + 40×40 + 20×20) × 3 anchors = 19200 + 4800 + 1200
- 每行 10 个值：`[cx, cy, w, h, obj_conf, cls0, cls1, cls2, cls3, cls4]`

**后处理步骤：**
1. 遍历 25200 行
2. 计算每个框的 `score = obj_conf × max(cls_scores)`
3. 过滤 `score < conf_threshold`（默认 0.25）
4. 对每个通过置信度阈值的框，取 `class_id = argmax(cls_scores)`
5. 将 cx,cy,w,h → x1,y1,x2,y2（绝对坐标 /640）
6. 按类别分组做 NMS（IoU 阈值默认 0.45）
7. 输出归一化到 [0,1] 的 `x, y, width, height, confidence, class_id, label`

### 4. .om 模型加载方案

在 `EntryAbility.onCreate()` 中：
1. 检查沙箱 `files/` 目录下是否已有 .om 文件
2. 如没有，从 `resourceManager.getRawFileContent()` 读取 .om
3. 写入到 `context.filesDir` 下
4. 将沙箱路径通过 NAPI 传给 C++ 层

### 5. UI 设计改进

- 标题区：应用名称 + 设备信息
- 图片区：Stack(Image + Canvas) 叠加，640×640 或自适应
- 结果区：检测框列表（类别 + 置信度），每类用不同颜色标记
- 操作区：选择图片按钮 + 推理状态指示
- 多类别不同颜色边框 + 标签

## Risks And Compatibility

1. **CANN SDK 环境依赖**：C++ 代码编译和运行需要板端安装 CANN toolkit，如果开发机没有交叉编译环境则 C++ 部分无法编译
2. **.om 模型格式**：必须与昇腾 310B 的 CANN 版本匹配，版本不一致会导致加载失败
3. **YOLOv5s 输出格式**：`[1, 25200, 10]` 假设了 5 类 + 5 bbox params 的布局，如果实际 .om 的输出 transpose 过（如 [1, 10, 25200]），后处理需要相应调整
4. **RGBA→RGB 转换**：在 C++ 端做通道分离和归一化，需确保字节序正确（HarmonyOS 使用 RGBA 还是 BGRA）
5. **内存对齐**：ACL 的 aclrtMalloc 对内存对齐有要求，需确保输入 buffer 大小与模型输入一致

## Rollback

1. 所有修改前的原文件已在 git 中有记录，可通过 `git checkout -- <file>` 回滚
2. 如 C++ 编译失败（缺少 CANN SDK），可临时将 hello.cpp 中的推理调用替换为模拟数据返回，前端 UI 不受影响
3. 如 .om 文件不存在，EntryAbility 中的拷贝逻辑会优雅失败，前端提示"模型未就绪"
